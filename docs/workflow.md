# 多对话协作工作流

本文记录 wxarticle 后续迭代的协作方式。目标是让需求、开发、修 bug、发布部署都能闭环，不依赖某一个长对话的隐性记忆。

## 基本角色

当前推荐保留 5 类对话：

```text
陈序：项目总控对话
林蓝：产品需求 / PRD 对话
周衡：架构重构 / 稳定性增强对话
沈修：功能开发与 Bug 修复对话
顾舟：GitHub 发布 / 部署运维对话
```

原始 MVP 对话命名为“袁启”，作为初代员工和历史档案保留，不再承接新任务。需要追溯早期决策时再回去查询。

## 新需求怎么走

新需求默认先发给项目总控对话，也就是陈序。

推荐提问格式：

```text
我有一个新需求：
【描述需求】

请你判断应该交给哪个专项对话，并拆成可执行任务。
```

总控判断规则：

- 需求还模糊：交给林蓝。
- 需求明确、属于普通小中型功能：交给沈修。
- 涉及鉴权、任务系统、数据库、模块拆分、AI client、核心稳定性：交给周衡。
- 涉及 commit、push、部署、服务器、发布检查：交给顾舟。

## 功能开发后的 bug 怎么走

如果 bug 出现在刚开发的新功能里，优先回到原开发对话反馈。这个对话拥有最近上下文，应该负责把功能推进到验收通过。

反馈格式：

```text
这是刚才这个功能的验收反馈，有 bug，请你先排查并修复。

现象：

复现步骤：
1.
2.
3.

期望结果：

实际结果：

补充信息：

要求：
- 先判断是不是本次改动引入
- 能在当前功能范围内修就直接修
- 如果涉及架构、部署或需求不清，不要硬改，输出交接给总控
- 修完后给出修改文件和验证结果
```

同一个功能在验收没通过前，不要让该对话继续接新需求。

## 线上或历史 bug 怎么走

如果 bug 不是刚开发功能产生的，或者 Gio 不确定属于哪里，先发给项目总控对话。

总控会判断：

- 小范围问题：交给沈修。
- 涉及架构边界：交给周衡。
- 涉及服务器、网络、计划任务、GitHub：交给顾舟。
- 涉及需求定义不清：交给林蓝。

## 避免 bug 相互影响

功能开发与 Bug 修复对话必须遵守：

- 每次只处理一个清晰任务。
- 开始前说明本次预计涉及哪些文件。
- 不顺手重构。
- 不同时让多个对话改同一批文件。
- 涉及 `src/formatter.py`、`src/web_app.py`、`src/main.py`、`src/article_generator.py` 时要特别谨慎。
- 两个 bug 都影响同一个模块时，排队处理，不并行处理。
- 修完一个 bug 并验证后，再开始下一个。

开始前建议声明：

```text
本次只处理：
预计涉及文件：
不碰的范围：
验证方式：
```

## 任务结束必须交接

每个专项对话完成任务后，都要使用 `docs/handoff_template.md` 的格式输出总结。

项目总控对话根据交接内容决定是否更新：

```text
docs/project_status.md
docs/roadmap.md
docs/known_issues.md
docs/changelog.md
docs/deployment.md
```

## 必须先确认的操作

以下操作不能由任何对话自行执行：

- 删除文件、目录、云端数据或 git 历史。
- 修改 `.env`、密钥、token、账号配置。
- 修改 CI/CD、部署配置、生产环境配置。
- 数据库 schema 变更、数据迁移、批量改数据。
- `git push`、`git rebase`、`git reset --hard`、强制推送、发布 PR。
- 安装新的全局依赖，修改系统配置，开启公网暴露服务。
- 部署到生产、npm publish、发文章、发邮件、发消息。

## 一句话原则

新需求先找总控；刚开发出的 bug 找原开发对话；不确定归属的 bug 找总控；发布和部署永远单独收口。
